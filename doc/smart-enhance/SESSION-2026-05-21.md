# PromptPro Smart Optimizer 实施会话记录

**日期**：2026-05-21
**主题**：按 `doc/smart-enhance/IMPLEMENTATION.md` 实施智能提示词优化器，并修复联动问题

---

## 一、需求来源

`doc/smart-enhance/IMPLEMENTATION.md` 要求实现一个纯前端的智能提示词优化系统：

- 无 API、无第三方库
- 将模糊的用户输入转化为清晰、自然的 AI 友好提示词
- 默认输出必须是自然语言（像真实用户提问），禁止 `#角色` / `#任务` / `#背景` / markdown 标题等 System Prompt 风格
- 内部允许有 `role / constraints / structure`，但 `humanizePrompt()` 必须将其转换为自然表达
- 三种模式：`light`（仅润色，简短）/ `smart`（结构 + 自然语言，默认）/ `pro`（可用角色和框架，但仍保持自然）
- 8 个分类：`content / coding / work / learning / life / travel / finance / general`，每个分类有独立 skeleton
- 加权关键词匹配做分类
- 自动推断上下文（platform / tone / audience / outputStyle），不向用户追问
- 6 个核心函数：`detectCategory / inferContext / buildPrompt / humanizePrompt / smartOptimize / quickEnhance`

## 二、实施过程

### 1. 第一次生成 `smartOptimizer.js`

在 [src/content/smartOptimizer.js](../../src/content/smartOptimizer.js) 创建了完整文件，包含：

- `categoryRules`：8 个分类的关键词、权重、骨架
- `platformRules / toneRules / audienceRules / outputStyleRules`：上下文推断规则
- `implicitPlatformRules`：隐含平台推断（如"减肥文案"→小红书）
- 6 个导出函数 + 内部辅助函数（`humanizeLight / humanizeSmart / humanizePro / polishInput / fillSkeleton / inferRole / adaptForModel`）
- ES Module 语法、中文注释

### 2. 第二次按更严格规则重写

补充 Final Output Rules（最高优先级）后重新执行：

- 三模式各自独立的 humanize 实现
- Light 模式输出风格示例：`请帮我写一篇减肥营销文案，语言自然真实，适合小红书风格，不要太像广告。`
- Smart 模式：使用骨架开场 + 自然要求列表，不用 `#` 标题
- Pro 模式：自然段落引入角色（`请你以XXX的身份来回答我的问题。`），不用 `#角色` 格式

### 3. 关键问题：新代码未生效

用户反馈"还是修改前的效果"。检查代码路径后发现：

- `smartOptimizer.js` 创建了，但**项目从未引用它**
- 应用实际调用的是 [src/content/optimizer.ts](../../src/content/optimizer.ts) 中的 `localOptimize()`，那是旧的 `# 角色 / # 任务` 模板
- 隐藏标签页路径用的是 `@/shared/constants` 里的 `META_PROMPT`，也是旧的 CRAFT 框架风格

**修复**：

1. 把 `optimizer.ts` 完全用 smartOptimizer 的逻辑改写（保留 `localOptimize` 和 `buildOptimizePrompt` 两个导出名以便兼容）
2. `localOptimize` 默认走 `smart` 模式，输出自然语言
3. 新增自然语言风格的 `META_PROMPT_NATURAL`，用于隐藏标签页路径
4. `background/index.ts` 改为引用 `buildOptimizePrompt` 而非旧 `META_PROMPT`
5. `content/index.ts` 删除未使用的 `buildOptimizePrompt` 引入

构建通过：`tsc --noEmit` 和 `vite build` 均无错误。

### 4. 关键问题：次数没有变化

用户反馈"优化成功之后，次数没有变化"。排查 `incrementUsage` 调用链：

- `background/index.ts` 的 `handleOptimize` 在隐藏标签页路径成功时会调用 `incrementUsage()`
- 但隐藏标签页路径**经常失败**（启动慢、平台拦截等），content script 走 fallback 路径用 `localOptimize`
- **fallback 路径从未调用 `incrementUsage()`**

**修复**：在 [content/index.ts](../../src/content/index.ts) 的 `else` 和 `catch` 两个 fallback 分支都加上 `await incrementUsage()`，保证无论走哪条路径，优化成功都计数。

## 五、待办与备注

- 隐藏标签页优化路径（`optimizeViaHiddenTab`）目前发给目标 AI 的指令已经是自然语言风格的 `META_PROMPT_NATURAL`，但实际成功率受平台 DOM 变化影响，仍以本地 `localOptimize` fallback 为主
- `MAX_FREE_DAILY_USAGE = 10`，重置逻辑基于 `lastResetDate`，跨天自动重置
- 如果未来要恢复隐藏标签页路径的稳定性，需要更新各平台适配器（chatgpt/claude/gemini/deepseek/doubao）的 DOM 选择器
